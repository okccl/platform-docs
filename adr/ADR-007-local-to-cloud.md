# ADR-007: Local→Cloud 移行パス設計方針

## Context

現在の職場では Tanzu 基盤を運用しているが、Broadcom による VMware 買収後の
ライセンス料高騰を受けて、GCP への移行を真剣に検討した経緯がある。
最終的にはコストの問題でオンプレの OpenShift に落ち着いたが、
その検討過程で「何をクラウドのマネージドサービスに置き換えるか」
「何をそのまま持ち込むか」という切り分けの難しさを経験した。

特に DB（PostgreSQL / Crunchy Data）周りの検討を担当し、
マネージドサービスへの移行ではバージョン追従の頻度やデータ移行の方法、
Monitoring 系ツールの UI 変更によるユーザーへの影響など、
技術面以外のコストも含めた判断が必要だということを実感した。

このポートフォリオでは Phase 10 までのローカル（k3d）環境を
Phase 13（EKS）・Phase 14（GKE）でクラウドに展開する。
その際の設計方針を事前に整理しておく。

---

## 移行対象の分類

### マネージドサービスに置き換えるもの
| コンポーネント | ローカル | AWS | GCP |
|---|---|---|---|
| Kubernetes | k3d | EKS | GKE |
| PostgreSQL | CNPG（self-managed） | RDS または CNPG on EKS | Cloud SQL または CNPG on GKE |
| オブジェクトストレージ | MinIO（Docker） | S3 | GCS |
| Secret 管理 | SOPS × Age | AWS Secrets Manager（ESO 経由） | GCP Secret Manager（ESO 経由） |
| 証明書 | cert-manager（自己署名） | ACM | Google-managed SSL |

### そのまま持ち込むもの
| コンポーネント | 理由 |
|---|---|
| ArgoCD | GitOps フローの一貫性を維持。クラウドネイティブの CD ツールより成熟 |
| Prometheus / Grafana | 既存ダッシュボードの継続利用。マネージドへの移行はユーザーへの説明コストが発生する |
| Cilium | EKS/GKE でも採用可能。kube-proxy 置き換えの効果をクラウドで検証する |
| Crossplane | provider-aws / provider-gcp でクラウドリソースを GitOps 管理する |
| KEDA / Argo Rollouts | デプロイ戦略・スケーリング戦略はクラウド非依存で継続 |

## 設計上の判断

### PostgreSQL の扱い

ローカルでは CNPG による self-managed HA 構成を採用しているが、
クラウドでは RDS / Cloud SQL（マネージド）との比較検証を行う。

マネージドサービスのメリットはバックアップ・フェイルオーバー・パッチ適用の自動化だが、
マイナーバージョンのサポート終了に伴う強制アップグレードへの追従コストが発生する。
また Crunchy から CNPG への移行経験から、PostgreSQL Operator の違いによる
移行コストは小さくないことがわかっている。
データ移行は pg_dump によるエクスポート・インポートを基本方針とする。

### Monitoring の扱い

Prometheus / Grafana はそのまま持ち込む。
AWS Managed Prometheus（AMP）や Google Managed Prometheus（GMP）との比較は
Phase 13・14 で検証するが、既存のダッシュボード資産が活用できることを優先する。
ユーザーにとっては使い慣れた Grafana の UI が変わることがコストになるため、
マネージドへの移行判断はユーザーへの説明コストも含めて評価する。

### コスト試算の難しさ

GCP 移行検討時の経験として、クラウドのコスト試算は前提次第で大きく変わる。
オンプレの運用コスト（人件費・保守費）をどこまで含めるか、
クラウドの割引契約（Reserved Instance など）をどう見込むかで結論が逆転することもある。
Phase 13・14 では OpenCost を導入し、実際の稼働コストを計測して比較する。

### ネットワーク設計

クラウドへの移行でネットワーク設計は最も環境依存が高い領域になる。
VPC 設計・セキュリティグループ・ロードバランサーの選択はクラウドプロバイダーの
制約に合わせて自然と選択肢が絞られてくるが、
オンプレとのハイブリッド構成が必要な場合は追加の設計コストが発生する。
このポートフォリオでは純粋なクラウド構成（ハイブリッドなし）を前提とする。

---

## Consequences

**ポジティブ:**
- ローカルで検証した GitOps フロー・デプロイ戦略・スケーリング設計をクラウドでも再利用できる
- マネージドとセルフマネージドの比較を実測値で評価できる
- OpenCost によるコスト可視化で、クラウド移行の判断材料を定量的に提示できる

**ネガティブ・トレードオフ:**
- 「そのまま持ち込む」コンポーネントが多いほど、クラウドネイティブの恩恵を
  十分に受けられない可能性がある。各クラウドのマネージドサービスとの比較は
  Phase 13・14 の検証結果を踏まえて継続的に見直す
- コスト試算は実測前の段階では不確実性が高い。Phase 13・14 完了後に改訂する予定
